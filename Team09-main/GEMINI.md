# {project maou}

## ゲームの概要

プレイヤーは**魔王**となり、次々と攻め込んでくる**勇者**を迎撃するコマンドラインベースのシミュレーションゲームです。
最大の特徴は、勇者やモンスターの思考・行動描写・ステータス生成に**LLM（大規模言語モデル）**を活用することです（現在はPhase 1としてモック稼働）。

倒された勇者はその無念を「遺言」として残し、その意志と戦闘経験が**次世代の勇者**へと継承（ステータス生成に影響）されます。魔王は配下のモンスターに自然言語で指示を出し、この終わりのない勇者の波を食い止めなければなりません。 
今回は勇者がフィールドモンスターに勝利→魔王には敗北して遺言をのこし表示するところまでを作成します。
JsonFolderにjsonファイルがあります。これはllmがこのjsonを各フェーズで生成したということとして開発を行います。

### ゲームループとフェーズ

ゲームは以下のフェーズで進行します。

1.  **初期化フェーズ（システム）**
    * フィールド（属性あり）、勇者、魔王、手持ちモンスター（5体）のステータスを読み込みます（LLM生成を想定したJSONファイルからのロード。今回はJsonFolder/maou,yuusya,field,monster.json）。
    * **属性ボーナス**: キャラクタの属性（炎/水/草）とフィールドの属性が一致した場合、HP・攻撃力・守備力が **1.1倍** に補正されます。

2.  **魔王フェーズ（ユーザー入力）**
    * ユーザーには「魔王ステータス」「フィールド名」「勇者ステータス」「モンスター5体のステータス」が開示されます。
    * ユーザーはモンスターを **2体** 選抜します。
    * 選抜した各モンスターに対し、**自然言語で「助言」や「命令」** を入力します。

3.  **反応フェーズ（内部処理）**
    * モンスターの性格と魔王の命令を組み合わせ、モンスターの「感情」や「覚悟」を更新します（今回は性格文字列への固定マージで代用。jsonのpersonality.jsonがその命令による感情や覚悟を示していると考えてください。）。

4.  **戦闘フェーズ（自動処理）**
    * 勇者 vs 選抜モンスター2体（＋フィールド補正）で戦闘を行います。
    * 戦闘経過と結果が表示されます（今回はプログラム内で定義された固定の描写テキストを表示）。

5.  **結果・継承フェーズ**
    * **勇者が勝利した場合**: 最終防衛ライン「魔王城」へ移動し、再度戦闘フェーズを行います。そこでも敗北すればゲームオーバーです。重要ー今回は起きません！
    * **勇者が敗北した場合**: 勇者は「遺言」を残して散ります。この「遺言」と「戦闘描写」が次世代の勇者生成の種（シード）となり、次のウェーブが始まります。

## 開発条件
* **言語**: Python 3.13
* **環境管理**: `uv`
* **開発手法**: テスト駆動開発 (TDD)
* **UI**: 現状はコンソール（CLI）。将来的には Pygame を使用。
* **LLM連携**: Phase 1では行わない（APIコール部分はモック化）。

## Python環境の管理
uvを使用してpython環境を管理しています。

パッケージの追加方法
```bash
uv add パッケージ名
```

## 補足情報
本プロジェクトは将来的なLLM連携を前提としていますが、まずはロジック検証のために以下のルールで実装します。

入力データ（JSON）:

勇者、モンスター、フィールドの初期ステータスは、LLMが生成したと仮定した JSONファイル として data/ ディレクトリ等に配置し、それを読み込みます。

スキーマは本番同様に定義します。

出力データ（プログラム内保持）:

戦闘描写（テキスト）や勝敗結果、モンスターの反応などは、JSONファイルにはせず、プログラム内のクラスや定数としてハードコード します。

これにより、将来 MockBattleService を LLMBattleService に差し替えるだけで済む設計にします。

仕様は spec ディレクトリにマークダウンファイルとして保存してください。 実装する前にタスク分解を行い、実装方針を固めてください。作成したタスクは Walkthrough.md に記載して進捗を管理してください。
